  <action>
Create the SQL migration file that provisions the Supabase Storage "avatars" bucket and its four RLS policies.

Content to write:

```sql
-- ============================================================
-- Storage: avatars bucket
-- Public bucket for user profile photos.
-- Each user's avatar stored at {user_id}/avatar.jpg
-- ============================================================
INSERT INTO storage.buckets (id, name, public, file_size_limit, allowed_mime_types)
VALUES (
  'avatars',
  'avatars',
  true,
  1048576,  -- 1MB max
  ARRAY['image/jpeg', 'image/png', 'image/webp']
)
ON CONFLICT (id) DO NOTHING;

-- Authenticated users can upload to their own folder only
CREATE POLICY "users can upload own avatar"
  ON storage.objects FOR INSERT
  TO authenticated
  WITH CHECK (
    bucket_id = 'avatars'
    AND auth.uid()::text = (storage.foldername(name))[1]
  );

-- Anyone can read public avatars
CREATE POLICY "avatars are publicly readable"
  ON storage.objects FOR SELECT
  TO anon, authenticated
  USING (bucket_id = 'avatars');

-- Users can update their own avatar (upsert support)
CREATE POLICY "users can update own avatar"
  ON storage.objects FOR UPDATE
  TO authenticated
  USING (bucket_id = 'avatars' AND auth.uid()::text = (storage.foldername(name))[1]);

-- Users can delete their own avatar
CREATE POLICY "users can delete own avatar"
  ON storage.objects FOR DELETE
  TO authenticated
  USING (bucket_id = 'avatars' AND auth.uid()::text = (storage.foldername(name))[1]);
```

Note: The `storage.foldername(name)[1]` helper extracts the first path segment from `{user_id}/avatar.jpg` — this scopes upload permission to the user's own folder.

Create two read-only GET route handlers for the profile tabs. Both require auth (401 for guests). Both use the SSR Supabase client.

**`frontend/app/api/profile/liked/route.ts`:**
- Auth check via `getUser()` — 401 if not signed in
- Query: `supabase.from('video_likes').select('created_at, videos(id, headline, video_url)').eq('user_id', user.id).order('created_at', { ascending: false })`
- The `videos` table does NOT have a `thumbnail_url` column (confirmed from /api/today/route.ts — only id, position, headline, source_url, video_url, duration, like_count, comment_count). Use `video_url` for thumbnail display via the VideoGrid component.
- Map the response to flatten: return `{ videos: data.map(row => ({ ...row.videos, likedAt: row.created_at })) }` — omit rows where `row.videos` is null
- Return `{ videos: [] }` if no likes yet (not an error)

**`frontend/app/api/profile/saved/route.ts`:**
- Same pattern but query `video_bookmarks` table instead of `video_likes`
- Map: `{ videos: data.map(row => ({ ...row.videos, savedAt: row.created_at })) }`
- Return `{ videos: [] }` if no bookmarks yet

Both: use `NextResponse.json(...)` for all responses. Handle DB errors with status 500.

Important: The Supabase embedded join syntax `.select('created_at, videos(id, headline, video_url)')` works because `video_likes.video_id` references `public.videos.id`. Supabase infers the join automatically.
